# uni-app 上线审计

> 说明
> 1. 本文是上线审计陪跑文档，不再单独定义产品范围。
> 2. 页面范围、角色边界、状态机与渠道能力以 [`docs/spec/PRODUCT-SPEC.md`](./docs/spec/PRODUCT-SPEC.md) 和 [`docs/spec/DOMAIN-API-SPEC.md`](./docs/spec/DOMAIN-API-SPEC.md) 为准。

## 目标

这份审计不是为了证明“页面能打开”，而是为了判断“这季节能不能上线可用”。当前判断口径固定为 3 条：

1. 主链路能跑通：浏览、搜索、详情、下单、支付、订单追踪、售后、消息、内容浏览、内容发布、师傅履约。
2. 失败场景能恢复：弱网、支付取消、返回重进、角色切换、空数据、无地址、无结果时都有明确下一步。
3. 页面可信：不再出现乱码、假入口、看不懂的状态文案，关键动作有解释，关键页面有空态和兜底。

## 当前已补齐的 P0

### 1. 文字可读和入口可信

- `pages.json` 已统一成可读中文标题和 Tab 文案。
- 首页生态位、小应学堂、商城、圈子、消息中心都已接真实路由，不再用“即将开放”硬占位。
- 登录、定位、优惠券、地址、下单、订单、售后、消息、圈子、学堂、商城、师傅端页面的用户可见乱码已清理。

### 2. 主交易链更像正式产品

- 商品/服务详情页已重排为“这是什么、价格是什么、下单后会发生什么、凭什么信任”。
- 下单页已补地址、时间、备注、支付说明、草稿自动保存。
- 订单详情页已改成“当前状态、下一步、现在能做什么”的结构，并承担继续支付、售后、催单、沟通入口。
- 轨迹页已从字段堆砌改为履约追踪视图，空轨迹和媒体文件都有兜底说明。

### 3. 支持链路和恢复链路已补

- 搜索页已补空结果、热词回填、清空和加载态。
- 圈子发布和下单备注已支持草稿恢复，减少误返回导致的内容丢失。
- 地址列表/编辑、定位页、优惠券页都补了空态和下一步动作。
- 消息中心、聊天页、角标同步、已读回写和断线重连已接通。
- 请求层已补 token 失效、弱网重试、追踪 ID 和记忆型错误提示。

### 4. 师傅端从“能点”推进到“敢用”

- 抢派单大厅会明确告诉师傅先看距离、地址和预估收入，再决定是否抢单。
- 工作台已按状态给出“现在建议你做什么”，并把签到、施工、报价、现场照片、完工提交分开讲清楚。
- 工作台补了离线队列说明和自动/手动补发，减少弱网下的恐慌。
- 钱包页已能说明可结算余额、冻结保证金、今日入账和流水的含义。
- 接单设置页已从单纯表单改成“听单、距离、隐私号为什么重要”的配置页。

## 从用户、UI、产品三条线反推出来的隐含需求

### 用户视角

用户真正关心的不是接口通不通，而是：

1. 这个服务或商品到底适不适合我当前的问题。
2. 我下单之后会发生什么，多久有人响应，钱付出去以后会不会石沉大海。
3. 如果支付失败、误返回、网络差，之前做的操作会不会白做。
4. 出问题时我去哪里找人，客服、售后、消息是不是同一套入口。
5. 这个页面像不像真产品，能不能让我放心继续点下一步。

因此这季上线必须坚持：

- 关键页面先讲清“下一步会发生什么”。
- 关键动作必须能恢复，不能一错就让用户从头重来。
- 没有专属会话时也要给用户客服兜底，而不是直接卡死。

### UI 视角

上线前最容易伤害信任的不是功能缺失，而是结构混乱：

1. 一屏看不懂状态和主动作。
2. 空态只有“暂无数据”，没有下一步。
3. 主按钮和次按钮层级不清。
4. 页面文案像半成品，用户一眼就会怀疑平台是否可靠。

当前 UI 处理原则：

- 一屏先回答“是什么、状态如何、现在该做什么”。
- 空态必须解释原因并给下一步。
- 高风险动作要给清楚的提示，而不是只给一个冷冰冰按钮。
- 交易页和履约页优先保障可读、可懂、可恢复，再谈装饰。

### 产品视角

真正的上线风险往往不是“少一个页面”，而是“链路断在看不见的地方”：

1. 支付失败后必须回到订单详情继续处理，而不是重走下单。
2. 城市、地址、首页内容、搜索、圈子、下单必须共享同一套位置上下文。
3. 角色切换不只是改 UI，还要让资料、token、入口行为一致。
4. 师傅弱网下的签到、状态推进、媒体上传必须自动补发，否则履约链不可信。
5. 钱包没有自助提现时，页面必须明确承担“结算透明”而不是伪装成功能完整。

## 当前仍需处理的 P0

这些仍然是本季上线前必须明确收口的事情：

1. 真实编译和真机冒烟还没做。
   现在做过的是源码解析，不是 HBuilderX 或微信开发者工具里的真实启动链。
2. 需要跑一次完整人工主链冒烟。
   C 端至少跑：首页 -> 搜索 -> 详情 -> 下单 -> 支付取消/失败 -> 订单详情 -> 消息 -> 售后。
3. 师傅端需要跑一次完整履约冒烟。
   抢派单 -> 接单 -> 出发 -> 到场签到 -> 施工前拍照 -> 增项报价 -> 完工拍照 -> 完工提交 -> 钱包查看流水。
4. 历史页面和未引用文件要继续清理。
   这不是审美问题，而是避免后续维护时又误引用回旧页面。

## 当前仍需处理的 P1

1. 售后页还没有补图片、录音等证据材料上传。
2. 搜索页还没有补历史记录和最近浏览联动。
3. 商城页还没有补筛选、排序、库存售罄态。
4. 圈子和学堂还没有把分享、举报结果反馈、发布成功回流做完整。
5. 钱包端内自助提现仍未开放，需要明确排期或继续维持“后台审核结算”。
6. 师傅端还缺“从工单直接进入当前聊天会话”的强关联入口。

## 建议的上线冻结范围

为了这季节真正能上线，不建议再横向加功能，建议冻结为下面这版：

- C 端重点保交易闭环、消息闭环、售后闭环、内容浏览闭环。
- 师傅端重点保抢单、履约、弱网补发、钱包透明、接单设置。
- Admin 和后端重点保首页运营、订单状态、支付退款、结算、内容审核。

超出这条线的功能，如果不能直接提升成交、履约成功率或售后恢复能力，建议延后。

## 验收口径

### C 端

- 首页进入详情、搜索、商城、学堂、圈子都不出戏。
- 下单前用户知道地址、时间、价格、支付方式和后续流程。
- 支付失败后能回到订单详情继续支付或处理，不会丢单。
- 订单详情能承担继续支付、催单、售后和沟通。
- 售后提交后能回订单详情看到变化。

### 师傅端

- 抢单、状态推进、签到、上传凭证、查看钱包、修改接单设置都能独立走通。
- 弱网下的离线队列能自动补发，不需要师傅理解系统机制才敢继续操作。
- 工作台每个关键动作前，师傅都看得懂“为什么现在该点这个”。

### 运营面

- 首页、学堂、圈子、商城入口都必须由真实数据驱动。
- 不允许再依赖前端硬编码假入口、假文案、演示态占位。
