# 「呼之应」O2O本地服务平台 - 全局产品需求与开发白皮书 (PRD & Architecture V2.0)

## 一、 产品定位与业务愿景

* **产品名称：** 呼之应
* **Slogan：** 一呼百应 · 专业到家
* **业务模式：** 软硬结合 + O2O本地生活综合服务。以家电维修、家政保洁、上门安装等低频刚需服务为入口，通过“全屋金管家”和“小应商城/社区”向高频消费和会员制转化。
* **双边市场模型：** C端（消费者）下单与评价闭环 + B端（服务提供者）抢接单与履约打卡闭环 + Admin端（平台方）调度与仲裁。

---

## 二、 核心技术栈架构 (Technology Stack)

* **前端 (Frontend)：** uni-app (Vue3) 跨端框架，首发微信小程序/App。
* **后端 (Backend)：** Java (Spring Boot 微服务架构)。
* **数据层 (Database/Cache)：** MySQL (持久化) + Redis (高并发缓存、订单超时、Geo 地理位置计算) + ElasticSearch (商品/服务的高效检索)。
* **第三方服务 (Third-Party)：**
  * **地图/定位：** 腾讯地图/高德地图 API（坐标拾取、距离计算、路线规划）。
  * **云存储：** 阿里云 OSS / 腾讯云 COS（图片、履约视频存储）。
  * **通讯：** 虚拟隐私号码服务（保护双方隐私）、WebSocket/第三方 IM（在线客服与聊天）。

---

## 三、 C端 (消费者端) 需求与页面清单

这是用户的核心动线，解决“找服务 - 下单 - 履约追踪 - 售后评价”的闭环。

### 1. 首页与导购模块 (Home & Navigation)
* **`pages/index/index` (首页核心)：**
  * **顶部动态 LBS：** 实时定位，动态渐变吸顶搜索栏。
  * **极速报修卡片 (T0转化)：** 高转化 C 位，雷达动效，点击唤起语音/图片快捷派单。
  * **金刚区导航：** SVG 双色图标大类入口（家电清洗、专业维修等）。
  * **SVIP金管家入口：** 承接高客单价包年服务，流光动效引导。
  * **生态便当盒：** 引流至“小应学堂”、“小应商城”、“同城圈子”。
  * **双列瀑布流推荐：** 猜你喜欢，动态高度计算，触底加载，沉浸式浏览。
* **`pages/location/select` (地址/城市切换)：** 列表页，支持拼音检索、历史城市记录、GPS 重新定位。
* **`pages/search/index` (搜索中枢)：** 历史记录标签、热门搜索（后台配置）、联想词输入提示。
* **`pages/category/index` (全部分类)：** 左侧大类 ScrollView 与右侧子类/商品流联动。

### 2. 服务详情与转化模块 (Service & Conversion)
* **`pages/goods/detail` (服务详情页)：**
  * 首屏：多图轮播/履约视频。
  * 价格透明化模块：基础服务费 + 预估上门费 + 详细计价规则（弹窗）。
  * 保障与背书模块：不修不收、迟到赔、质保期。
  * 服务流程图文富文本展示。
  * 用户评价摘要（突出带图评价）。
  * 底部悬浮栏 (Dock)：联系客服、分享、加入清单、立即预约按钮。
* **`pages/goods/comment` (全部评价页)：** 支持好/中/差评标签切换，支持图片放大预览。

### 3. 订单与支付模块 (Order & Checkout)
* **`pages/order/checkout` (预约下单确认)：**
  * **地址选择模块：** 必须强制要求选择精确的门牌号地址。
  * **预约时间选择 (Time Slot)：** 底部弹出日期/时间段（如：今天 14:00-16:00），支持“极速上门”选项。
  * **故障描述与凭证：** 支持文本、图片、录音上传。
  * **费用明细与支付渠道：** 微信/支付宝唤起支付定金或预付款。
* **`pages/order/list` (订单列表)：** 分为全部、待付款、进行中、待评价、售后。
* **`pages/order/detail` (订单详情与跟踪)：**
  * **状态时间轴 (State Machine)：** 派单中 -> 师傅接单 -> 师傅出发 -> 师傅到达 -> 服务完成。
  * **动态加价确认确认单 (核心防坑)：** 师傅发起额外报价时，用户端弹出需确认的明细卡片，确认后追加支付。
  * **一键催单 / 取消订单：** 区分取消规则（派单前秒退，派单后需审核违约金）。
* **`pages/order/tracking` (地图轨迹监控)：** （高级功能）调用地图 SDK 实时查看师傅移动轨迹及预计到达时间 (ETA)。

### 4. 个人中心与支撑模块 (User Center & Support)
* **`pages/user/index` (个人中心首页)：** 整合订单控制台、资产中心（优惠券/余额）、SVIP状态、常用功能列表。
* **`pages/setting/address-list` & `address-edit` (地址管理)：**
  * 地图选点 (POI) + 门牌号详细补充。
  * 自动智能解析粘贴文本。
  * 默认地址设置。
* **`pages/message/chat` (在线沟通)：** 订单关联的 IM 聊天，支持发图和语音。
* **`pages/marketing/coupon` (优惠券中心)：** 红包和优惠券的发放与核销展示。
* **`pages/auth/login` (鉴权模块)：** 微信快捷授权、手机号验证码、免密一键登录。
* **`pages/apply/master` (师傅入驻申请)：** 资质上传、工种选择表单，C 转 B 桥梁。

## 四、 B端 (师傅/技师端) 需求大纲 (Master App)

为保证服务质量，必须配备专用的师傅端（可以是独立的小程序或 App），作为平台履约的抓手。

### 1. 抢派单大厅 (`pages/master/dispatch`)
* 展示附近的订单（距离排序），可抢单或接收平台强派单。
* 订单详情概览（故障描述、预估收入、距离等，隐藏用户真实号码）。

### 2. 工作台与标准打卡流程 (`pages/master/workbench`)
* 操作节点：【开始出发】->【到达现场（LBS校验）】->【服务前拍照】->【提交维修方案及新增报价（如有）】->【服务完成拍照】。

### 3. 动态报价单系统 (Quotation System)
* 师傅在现场根据实际情况，选择配件 SKU 或手动输入增项金额，发送给用户端确认。

### 4. 钱包与结算 (`pages/master/wallet`)
* 收入明细流水，提现申请，保证金管理。

### 5. 排班与接单设置 (`pages/master/setting`)
* 一键开启/关闭听单，设置最大接单距离。

## 五、 后台管理系统 (Admin Panel) 核心需求

平台的调度中枢，协调 C 端与 B 端，并处理客诉异常。

### 1. 订单调度中心 (Dispatch Center)
* 可视化地图派单（查看订单密集度和附近空闲师傅分布）。
* 人工干预改派、强制取消。

### 2. 师傅管理 (Master Management)
* 资质审核，实名认证，等级评定。
* 信用分与处罚机制（如：乱收费被投诉扣除信用分，限制接单）。

### 3. 财务与售后仲裁 (Finance & Arbitration)
* 退款审核。
* 客诉仲裁单：查看双方沟通记录、上传的图片，判定责任归属。

### 4. 服务类目与定价引擎 (Pricing Engine)
* 后台配置分类树，配置各服务的“基础价”、“指导价”。
* 配置时间系数（如：夜间服务费 +30%）。

## 六、 隐性机制与关键技术难点 (Crucial Mechanics)

在代码开发前，团队必须达成共识的业务难点：

1. **LBS 与服务边界计算：** 不能仅仅计算直线距离。需要结合高德/腾讯地图的区域多边形边界判断算法，确认用户地址是否在某个门店或师傅的履约范围内。
2. **状态机与并发一致性：** O2O 订单状态变化极快。比如：两个师傅同时抢一个订单，或者用户发起退款的同时师傅点击了“到达现场”。后端必须依靠 Redis 分布式锁和严谨的状态机（State Machine）防止脏数据产生。
3. **资金池与分账逻辑：** 如果不是自营师傅，用户的钱不能直接进入师傅账户。需遵守央行“二清”规定，采用微信分账/支付宝分账体系，订单确认完工并过完售后期后，自动分账到师傅账户。
4. **Token 防刷新拦截：** 前端需在 Axios/uni.request 拦截器中，处理好静默刷新 Token 的逻辑，保证用户在下单关键节点不会突然掉线。
5. **弱网容灾设计：** 师傅经常在地下室或封闭环境维修，网络极差。打卡或拍照上传的接口，前端需要做离线缓存重试机制。



# 「呼之应」O2O本地服务生态平台 - 全局产品与技术全景白皮书 (PRD & Architecture)

## 一、 核心战略与三维架构思维 (Core Strategy)

本项目打破传统 O2O 平台的工具属性，采用“生态化”架构，实现从低频维修到高频电商/社区的跨越。

### 1. 用户思维 (User Flow) - 需求急迫度分层
* **T0 级（急需型）：** 顶部最高视觉层级悬浮**【极速报修】**。雷达波纹强引导，跳过繁琐分类，直接语音/LBS派单解决焦虑。
* **T1 级（刚需型）：** 中部核心区部署**【金刚区大类】+【全屋金管家】**。承接明确的安装/保洁需求，并通过管家服务拉升客单价。
* **T2 级（闲逛型）：** 底部部署**【生态便当盒】+【双列瀑布流】**。以图文并茂的社区内容和五金耗材电商留存用户。

### 2. 产品思维 (Product Flow) - 流量分发闭环
* **主营业务：** 强 LBS 定位的同城上门服务（维修、保洁、安装）。
* **交叉引流：** 通过首页便当盒，向「小应学堂（培训）」、「小应商城（耗材电商）」、「同城圈子（社区）」分发流量，构建本地生活壁垒。

### 3. UI/UX 视觉思维 (Design System) - 科技信任感
* 彻底废弃传统家政的绿色/暖橙色系，重塑为**“科技 + 高端管家”**的视觉基调。
* 全面使用**高质量双色 SVG 图标**替代单色线框，保证 Retina 屏幕上的极致清晰。

---

## 二、 详细 UI 设计规范体系 (UI & Visual Guidelines)

### 1. 全局色彩系统 (Color Palette)
* **品牌主色 (Primary)：** `科技信任蓝 #2B5CFF`（用于全局重要按钮、图标主色、标签强调，传递极速与专业）。
* **主色辅助色 (Secondary)：** `浅冰霜蓝 #8AB4F8`（用于 SVG 图标的双色点缀、背景微光）。
* **高净值/VIP色 (Premium)：** `流沙金 #FFE0A3` 搭配 `深邃黑 #1C1E26`（专用于全屋金管家、SVIP 标识，拉开价格锚点）。
* **背景与底色 (Background)：** `冷净灰 #F4F6F9`（页面底色，衬托纯白卡片的边缘）。
* **功能色 (Functional)：** 价格红 `#FF4A4A`，警示橙 `#FF7D00`，安全绿 `#00B578`。

### 2. 材质与视觉特效 (Materials & Effects)
* **玻璃拟物化 (Glassmorphism)：** 大量使用 `backdrop-filter: blur(12px)` 和微透明背景（如 `rgba(255,255,255,0.2)`），打造悬浮感。
* **情绪化动效 (Motion)：**
  * 极速报修卡片：`radar-pulse` 呼吸波纹雷达动效。
  * VIP 金管家卡片：`glare-effect` 5秒周期的扫光溢彩特效。
  * 全局交互反馈：所有可点击卡片和按钮，必须带有 `transform: scale(0.96)` 的物理按压收缩反馈。

## 维度二：UI / UX 视觉设计视角 (UI & Visual)
*该视角明确了产品的视觉基因、交互规范和组件要求，供 UI 交互设计师和前端组件开发者使用。*

### 1. 全局设计语言与色彩规范
* **设计基调：** 摒弃传统家政的暖色调，重塑为“科技 + 高端管家”的信任感。
* **色彩系统：**
    * 品牌主色：`科技信任蓝 #2B5CFF`（核心按钮、强调标签）。
    * 主色辅助：`浅冰霜蓝 #8AB4F8`（背景微光、图标点缀）。
    * 高净值色：`流沙金 #FFE0A3` + `深邃黑 #1C1E26`（全屋金管家、SVIP 专属）。
    * 背景与底色：`冷净灰 #F4F6F9`（用于衬托白色卡片边缘）。
    * 功能警示色：价格红 `#FF4A4A`、警示橙 `#FF7D00`、安全绿 `#00B578`。
* **图标规范：** 彻底废弃单色线框，全面使用**高质量双色 SVG 图标**，确保多端高清渲染。

### 2. 材质与全局交互动效
* **卡片材质：** 广泛采用卡片式布局（圆角 `24rpx`），结合**玻璃拟物化 (Glassmorphism)**，使用 `backdrop-filter: blur(12px)` 与微透明背景打造悬浮感。
* **情绪化微动效：**
    * 所有可点击组件：必须附带物理按压反馈（`transform: scale(0.96)`）。
    * 极速报修卡片：`radar-pulse` 呼吸波纹雷达动效，强化 T0 转化。
    * 语音输入按钮：需具备录音波纹反馈。
    * SVIP 金管家卡片：5秒周期的 `glare-effect` 扫光溢彩特效。

## 维度三：前端工程化与技术实施视角 (Frontend Architecture)
*该视角梳理了前端的代码组织、极端场景容灾和底层优化策略，供前端开发团队作为核心参考。*

### 1. 前端基础架构
* **技术选型：** uni-app (Vue3) 跨端框架 + Pinia 状态管理，首发适配微信小程序与 App。
* **安全区适配：** 全局页面必须处理沉浸式底部逻辑，写入 `padding-bottom: env(safe-area-inset-bottom)` 避免被 iPhone 小黑条遮挡。

### 2. 核心技术痛点攻坚
* **无感知鉴权 (Token 静默刷新)：** Axios/uni.request 拦截器必须处理 401 Unauthorized，实现 Token 静默刷新或无缝引导登录（登录后原路重定向回详情页），防止支付链路中断。
* **弱网环境容灾 (离线打卡)：** 针对师傅常处于地下室等网络盲区，前端需实现打卡、拍照记录的**离线本地缓存机制**，网络恢复后自动断点重传。
* **长列表性能优化：** 首页 C 端双列瀑布流需实现动态高度计算、触底加载与防抖处理，必要时引入虚拟列表。
* **多媒体性能：** 服务详情页首屏若为履约视频，需实现自动静音预加载；图片上传需对接云存储实现 WebP 压缩。

## 维度四：用户体验与全链路动线视角 (User Journey)
*该视角模拟了双边用户的实际使用旅程，明确了每个关键节点的业务要求，供产品运营和测试人员做场景走查。*

### 1. C端消费者核心动线 (分层转化策略)
* **【发现与定位】基于意图分层流量：**
    * **进入 App：** 强制请求位置权限，顶部动态 LBS 展示所在小区，并据此决定可提供的服务与起步价。
    * **T0 级急需用户：** 视觉最高优先级点击【极速报修】，通过 AI 语音自动提取故障关键词或图片一键派单。
    * **T1 级刚需用户：** 通过中部【金刚大类】进入确定的服务（保洁/清洗），或点击【全屋金管家】购买大额服务。
    * **T2 级闲逛用户：** 浏览底部【生态便当盒】进入社区/商城，或刷瀑布流看师傅的施工案例对比图。
* **【详情与决策】打破电商逻辑：**
    * 强化非标服务展示：重点展示“基础检测费 + 维修耗材指导价”。
    * 强化信任背书：显眼位置展示不修不收、迟到赔款、平台实名认证、90天质保及保证金标识。
* **【下单预约】降低沟通成本：**
    * 地址选择：必须通过地图 POI 选点 + 手动精确门牌号（支持粘贴文本智能解析）。
    * 时间选择：底部弹窗选择“期望上门时间段”或直接勾选“极速上门”。
    * 定金支付：支持图文、录音描述故障，支付预付款锁定订单。
* **【履约与售后】核心防坑闭环：**
    * 履约监控：状态轴实时更新，可查看师傅地图移动轨迹与 ETA（预计到达时间）。
    * **动态加价防坑确认：** 现场如需增项，用户端需收到师傅发起的《增项报价单》（明细列表），强制用户弹窗点击确认后，方可追加支付并开工。
    * 售后/中心：可一键催单/取消（区分违约金规则），提供在线 IM 联系客服发送图文，支持发起乱收费举报退款。

### 2. B端技师履约核心动线 (Master App)
* **【接单入驻】C转B桥梁：** C端用户可通过 `apply` 模块提交资质认证入驻。
* **【抢派单大厅】信息过滤：** 基于 LBS 距离排序查看订单，系统实时语音播报。查看订单详情（隐藏真实用户手机号以防私下接单）。
* **【五步标准打卡】履约工作台（强执行环节）：**
    1.  点击【开始出发】。
    2.  到达现场进行【LBS 定位校验打卡】。
    3.  进行【服务前拍照】留存证据。
    4.  【动态报价】：若有新增项，选择配件 SKU 或手动输入金额发送至用户端确认。
    5.  完成服务进行【完工验收拍照】，系统自动生成《电子服务报告》推送给 C 端。
* **【钱包与排班】自我管理：**
    * 钱包中心查看分账流水，管理保证金，发起 T+N 提现。
    * 设置后台：一键开启/关闭听单模式，设置最大可接单半径距离。
