# 呼之应产品摘要

## 1. 产品定位

呼之应是一套覆盖用户端、师傅端、运营后台的本地生活服务平台，核心经营模式是平台自营撮合：

- 平台统一获客、定价、接单、收款、售后和结算。
- 师傅是平台履约方，不是独立商家。
- 商品单和服务单共用同一套用户、支付、消息、售后和后台运营体系。

## 2. 目标用户

- 家庭用户
  需要预约维修、安装、清洗、保洁、耗材购买和安装联动。
- 师傅
  需要稳定接单、明确工单、弱网可履约、补价留痕、结算透明。
- 平台运营
  需要调度订单、处理退款仲裁、维护内容与首页运营位、管理定价和师傅质量。

## 3. 当前版本定义

- 版本目标：真生产扩张版
- 发布策略：先按首批城市白名单上线，再复制城市运营模板
- 架构边界：不拆微服务，不引入多商家和加盟商体系
- 渠道策略：多端并行，但按小程序/H5/App 能力分层定义降级规则

## 4. 业务优先级

### P0：交易履约闭环

- 服务单：浏览、下单、预支付、派单/抢单、出发、到场、补价、补款、完工、售后、退款
- 商品单：浏览、下单、支付、发货、安装工单联动、完成/退款
- 师傅侧：抢单、工作台、签到、施工前后凭证、报价、完工、钱包流水
- 后台侧：调度、订单、财务、退款、仲裁、权限

### P1：内容与增长

- 首页运营位、公告、热词、推荐排序
- 小应学堂、同城圈子、优惠券、会员、搜索增长
- 内容审核、举报处理、活动联动

### P2：扩张能力

- CRM、经营分析、城市策略
- 更强监控、告警、自动化运维
- 更复杂营销和精细化推荐

## 5. 成功标准

### 工程标准

- `uni-app`、Admin、Spring Boot 根工程可稳定启动、测试和导出文档。
- 接口分组、页面路由、文档目录之间不存在版本漂移。

### 业务标准

- 三条主链至少各有一条真实可走通：
  - 用户下单到履约完成
  - 师傅接单到钱包流水入账
  - 后台调度到退款/仲裁处理完成

### 体验标准

- 关键页面先讲清“当前状态、下一步、可执行动作、失败后恢复方式”。
- 支付失败、弱网、地址越界、身份过期、重复点击都能得到业务可理解提示。

## 6. 核心指标

- 成交转化率
- 支付成功率
- 派单命中率
- 接单时长
- 准时到场率
- 完工率
- 退款 SLA
- 师傅活跃率
- 内容助攻转化率

## 7. 反目标

- 不做多商家入驻与商家自营
- 不做加盟商或多租户隔离
- 不做第三方 IM / 社区 SaaS
- 不把“演示可看”误当成“生产可运营”
- 不允许前端硬编码页面假入口、伪支付成功或绕过权限
