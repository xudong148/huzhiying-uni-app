# 呼之应规格包索引

本目录是呼之应项目的单一事实来源。凡是产品范围、页面规则、角色权限、状态机、数据结构、接口契约、后台能力、验收口径发生变化，必须先更新这里，再同步到代码与导出文档。

## 文档结构

- `PRODUCT-SUMMARY.md`
  执行层产品摘要，回答产品是什么、给谁用、先做什么、不做什么、如何判断是否达标。
- `PRODUCT-SPEC.md`
  主规格文档，覆盖用户旅程、页面清单、角色边界、UI 状态、异常恢复、后台流程、渠道能力矩阵。
- `DOMAIN-API-SPEC.md`
  领域与接口文档，覆盖实体模型、字段规范、状态机、接口分组、错误码、审计规则、文档同步规则。

## 使用规则

- `../project/PRODUCT-BRIEF.md` 只保留产品摘要入口，不再维护独立版本。
- `../project/PLAN.md` 只保留规格落地路线和执行顺序，不再承载完整业务规格。
- `../api/*` 是接口契约的离线导出，不是需求来源；接口分组变化后必须重新导出。
- `../project/UNIAPP-LAUNCH-AUDIT.md` 是上线审计陪跑文档，结论必须引用本目录的规格，而不是单独扩展业务范围。

## 当前基线

- 目标版本：真生产扩张版
- 经营模式：平台自营撮合、平台统一收款结算
- 终端策略：多端并行，按渠道能力分层，不允许静默失效
- 业务优先级：交易履约优先，内容生态服务转化和留存
- 技术架构：`uni-app + Vue Admin + Spring Boot 单体`

## 变更同步要求

1. 产品或流程变化：先改 `PRODUCT-SUMMARY.md` / `PRODUCT-SPEC.md`
2. 数据模型、状态机、接口变化：同步改 `DOMAIN-API-SPEC.md`
3. 后端接口分组变化：同步改 `OpenApiConfig`、测试和 `../api/*`
4. 页面范围变化：同步改 `pages.json`、主规格页面矩阵和上线审计
